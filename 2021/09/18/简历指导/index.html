就业阶段工作流程

简历→面试→入职→试用期→转正

简历的重要性：

  • 没有简历就没有面试
  • 简历的质量决定面试的数量

写简历的心法

一个字：猛！

整个简历充满有技术含量的技术点！给人一种厚重感！让 HR“不明觉厉”！

例:

  • 熟悉 Java 的 8 大基本数据类型
  • 熟悉 Java 的类加载机制、垃圾回收机制、内存管理模型

友情提示：简历和复习的关系

  • 正确的顺序是：先把简历写好，然后按照简历复习
  • 错误的顺序是：写简历的时候按现在“会”的写

写简历的注意事项

不要复制粘贴，要自己写

不要有技术性错误：硬伤

注意技术英文名称大小写

简历的结构

  • 个人信息：就业老师
  • 自我评价：就业老师
  • 技能清单：尽可能全面的覆盖学过的内容
    • 取：捡有技术含量的写
    • 舍：看上去很水的就不写了
  • 项目经验：把我们学习过程中项目里用到的技术点都体现出来

JavaSE

项目经验

项目描述

用平实的语言简单介绍一下这个项目即可。让读者能够知道这个项目是解决现实生活中的什么问题。在项目描述这个部分中，项目的架构可写可不写。要简洁明了，不要长篇大论。

众筹：在互联网上打造一个金融平台，让有志于创业的人在这个平台上可以募集到资金。而支持创业项目的用户可以得到相应的回报。

电商：互联网购物平台。进一步介绍商业模式，是 B2C、B2C、C2C 等等中的哪一种。再介绍一下销售的是什么产品，例如：综合性电商什么都卖、专注某一个细分市场（电子产品、服装、食品、体育用品……）。有些电商会有自己的特色，例如：社交电商。

在线：互联网学习平台。进一步介绍内容来源：平台自己生产内容还是商家入驻等等。

所以在简历上，可以有所区分：

  • 负责的模块：开发了该模块大部分功能
  • 参与的模块：开发了该模块较少部分功能

各个模块不必再展开介绍，因为一般来说看模块的名字就知道这个模块的功能。也不必写这个模块用到的技术，因为后面在技术描述中会专门罗列项目中用到的技术。、

结论：责任描述的部分就是简简单单的写模块名字列表即可。

技术描述

注意点 1：不要把项目 A 的技术描述直接复制到项目 B 里面。不同项目的技术描述中可以包含相同技术点，但是要结合各个项目的具体功能，进行不同描述。

  • 正确例子：
    • 项目 A：使用 Redis 缓存“众筹项目”数据。
    • 项目 B：使用 Redis 缓存用户登录的 token。
  • 错误例子：
    • 项目 A：使用 POI 将统计数据导出到 Excel 文件中。
    • 项目 B：使用 POI 将统计数据导出到 Excel 文件中。

注意点 2；让技术点和项目中的功能结合到一起。让技术的描述更饱满，能够体现出来是真的在项目中用到过。

  • 正确例子：
    • 项目 A：使用 Redis 缓存“众筹项目”数据。
    • 项目 B：使用 Redis 缓存用户登录的 token。
  • 错误例子：
    • 使用了 Redis。
    • 使用了 ElasticSearch。
    • 使用了 POI。
    • 使用了 Echarts。

OA

  • 单一架构：后台管理系统
    • 使用 Maven 作为构建和依赖管理工具
    • 使用 SSM 整合环境作为基础架构
    • 使用 MyBatis 的 PageHelper 插件实现分页
    • 在数据库表中使用 pid 字段建立记录之间的父子关系
    • 在页面上使用 zTree 展示树形结构
    • 使用 SpringSecurity 完成后台管理系统的权限控制
    • 使用 BCryptPasswordEncoder 实现带盐值的密码加密
  • 分布式架构：前台会员系统
    • 使用 SpringBoot+SpringCloud 搭建底层分布式开发环境
    • 调用第三方短信接口在注册用户时发送验证码短信
    • 调用阿里云的 OSS 对象存储服务存储用户发布项目时上传的图片
    • 调用蚂蚁金服的支付接口
    • 使用 SpringSession 解决分布式环境下 Session 共享问题，使用 Redis 作为 Session 库
    • 使用 swagger 生成接口文档
    • Redis 缓存对应用户登录状态的 token
    • 使用 Redis 临时保存项目发布过程中的数据，全部数据收集完毕保存到数据库

电商

  • 数据结构设计
    • spu+sku
    • 产品图片存储：FastDFS，使用 Nginx 做 Web 代理服务器接受访问
  • 缓存
    • 缓存商品数据提高查询效率
    • 分布式锁
      • 为了避免在 Redis 缓存服务器宕机后把 MySQL 服务器压垮，另外在其他 Redis 服务器中存储分布式锁，访问商品数据需要拿到锁，排队访问。
      • lua 脚本或者同步锁值判断防止并发时错误删除同步锁
  • 销售属性
    • 动态切换同类商品的解决方案
    • 销售属性静态化，生成销售属性 hash 文件存入浏览器，避免二次访问服务器
  • 搜索
    • 使用 ES 作为商品数据的搜索引擎
    • ES 集群搭建
    • ES 集群管理
    • ES 集群通讯
    • ES 数据结构
    • ES 数据导入查询等客户端 API
    • 中文分词器：IKAnalyzer
  • 搜索功能的解决方案
    • 关键词搜索
    • 平台属性过滤
    • 面包屑
    • 高亮显示、分页、排序、聚合等
  • 购物车解决方案
    • 分别在 Cookie、Redis、MySQL 中存储购物车数据
    • 用户未登录时在 Cookie 中存储购物车数据
    • 用户登录后要进行合并
    • Redis 中使用 hash 类型存储的
  • 异步服务和消息队列
    • 订单、支付、库存等系统在用户提交订单时通过消息队列启动异步服务
    • 用户登录时，启动异步服务，同步购物车、短信平台、站内信、日志等系统（相对我们自己来说属于其他开发人员负责的系统）
    • 使用延迟队列检查商品订单的支付情况
    • 在缓存中存入“交易码”，来保证结算不能重复提交，一次结算页面只能 提交一次订单
    • 死信队列，消息阻塞等情况处理（理解）
  • 支付
    • 调用支付宝的支付接口
    • 使用 rsa 非对称密钥的签名验证
    • 交易结果查询、支付状态查询、对账等（这些功能在电商系统中肯定要具备，但是不是由单个一个程序员完成）
  • 库存和物流业务对接
    • 库存和物流的调用关系（理解）
    • 拆掉业务流程（把购物车中的数据分配到不同仓库分别进行出库）
  • 多线程应用
    • 技术上使用 juc 中 FutureTask 创建多线程
    • 提交订单时用于验证和查询价格、库存、修改优惠券、满减资格状态等等
    • 对于过期、超时退单等无效订单集中处理
    • 新增或者修改商品信息以后，对相关商品的优惠、活动、券、折扣等服务调用
